## 2. 완료된 작업

### 2-1. 지오코딩 API 선정 (카카오 → VWorld 전환)

**최초 계획**: 카카오 REST API
- 정확도가 가장 좋다고 알려져 처음에 선택
- 발급받은 키가 **테스트앱**(개발 모드) 상태였음

**문제 발생**:
- 단일 호출 테스트는 정상 작동(200 OK)했으나, 6,690건 일괄 호출 시 97건(1.4%)만 성공 후 차단
- 응답 메시지: `"API limit has been exceeded"`
- 원인: 테스트앱은 일일 호출 한도가 매우 낮음. 일반 앱 전환은 도메인 등록 등 절차 + 비즈앱 전환은 사업자 등록증 필요

**대체 선택: VWorld API (국토교통부)**
- 일일 한도 40,000건 (개발용 키도 동일)
- 회원가입 후 즉시 발급
- 한국 도로명 주소 정확도 우수
- 6,690건 처리에 충분한 여유 (사용률 약 17%)

### 2-2. 아파트 좌표 변환 (지오코딩)

- 실거래 3개 파일 concat → 시군구 컬럼이 "서울특별시"로 시작하는 건만 필터링 (106,124건)
- 같은 단지는 좌표가 동일하므로 **`단지명 + 도로명_전체` 기준 unique 추출** → 6,690개 단지
- VWorld API로 도로명 주소(`서울특별시 + 구 + 도로명`) → 위경도 변환
- 처리 시간: 약 19분 (sleep 0.1초, timeout 5초)
- **성공률: 6,670 / 6,690 (99.7%)**

**왜 10.6만 건 전부가 아니라 6,690 단지만 지오코딩했나**:
- 실거래 데이터는 거래 1건 = 1행 구조라 같은 단지에서 발생한 거래가 수십~수백 번 반복됨
- 단지당 평균 약 16건 거래 (106,124 / 6,690)
- 같은 단지의 좌표는 항상 동일하므로 unique 처리 후 한 번만 변환해도 정보 손실 없음
- API 호출 약 **16배 절약** (10.6만 → 6,690건), VWorld 일일 한도 사용률 약 17%
- 처리 시간도 4~5시간 → 19분으로 단축
- 지오코딩 결과는 마지막에 **`단지명 + 도로명_전체` 기준 left join**으로 원본 10.6만 건 전체에 다시 매핑되므로, 최종 산출물의 모든 거래 행이 좌표를 가짐

### 2-4. 지하철역 거리 계산

- 각 단지 좌표 vs 783개 역 좌표 haversine 거리 계산
- 환승역 중복(128개)은 좌표가 거의 같아 최소값 산출에 영향 없음 → 별도 제거 없이 그대로 사용
- 출력 변수:
  - `최근접_역명`: 가장 가까운 역의 이름
  - `최근접_역_호선`: 해당 역의 호선 (환승역인 경우 가장 가까운 좌표의 호선)
  - `최근접_역_거리(m)`: 미터 단위 거리

**왜 서울 역만 거르지 않고 수도권 전체(인천·경기 포함) 그대로 유지했나**:
- 분석 대상은 서울 단지지만, 부동산 가치는 **행정 경계가 아닌 실제 접근성**에 의해 결정됨
- 서울 외곽 지역(강서·강동·송파·금천·은평구 등)에서는 인접 경기/인천 역이 서울 역보다 가까운 경우 다수
  - 예: 강서구 단지 → 김포골드라인, 공항철도 인접
  - 예: 강동구 단지 → 5호선 연장(미사·하남) 인접
  - 예: 송파구 단지 → 8호선 연장(별내·다산) 인접
  - 예: 강남 일부 단지 → 신분당선 연장(판교 방면) 인접
- 만약 서울 역만 남기면 외곽 단지의 "최근접 역"이 실제보다 멀리 잡혀 변수 왜곡 발생
- 어차피 거리 계산은 최소값만 사용하므로 서울 외 역이 후보로 들어가도 영향 없음 (대부분 단지는 서울 내 역이 최근접으로 선정됨)
- → 데이터 손실 없이 보수적으로 전체 수도권 역을 그대로 활용

### 2-5. 학교 거리/수 계산

- 학교 데이터에서 `소재지지번주소.str.startswith('서울')`로 서울 학교만 필터링
- `학교급구분` 컬럼 기준 초/중/고 분리
- 각 단지 좌표에서 학교별 haversine 거리 계산
- 출력 변수:
  - `최근접_초등학교` / `최근접_중학교` / `최근접_고등학교`: 학교명
  - `최근접_초_거리(m)` / `최근접_중_거리(m)` / `최근접_고_거리(m)`: 미터 단위 거리
  - `초등학교_수` / `중학교_수` / `고등학교_수`: 반경 1km 내 학교 개수

### 2-6. 원본 데이터에 변수 결합

- 6,690 단지 마스터 → 원본 106,124 거래에 `단지명 + 도로명_전체` 기준 left join
- 원본 컬럼 모두 보존 (전처리 파트에서 자유롭게 정제하도록)

### 2-7. 기준금리 컬럼 추가

- 한국은행 기준금리 데이터를 wide → long 변환 (월별 컬럼 → `계약년월` + `기준금리` 2컬럼)
- 연월 포맷 통일: `"2022/01"` → `202201` (실거래의 `계약년월` 컬럼과 매칭용)
- `계약년월` 기준으로 매핑하여 `기준금리` 컬럼 추가
- 결측 0건 (실거래 기간 2022.01~2024.12 전체가 금리 데이터 범위 내)
- **금리 가공(이동평균, lag, 변화율 등)은 본인 파트가 아닌 모델링/전처리 파트 책임으로 인계**
- 최종 컬럼 수: 36개 (원본 20 + 추가 16)

### 변수 생성 섹션 일부 (배점: 30%)

- 좌표 기반 거리 변수 생성 과정 (haversine 공식 사용 이유)
- 학교 수 변수의 학교급 분리 근거 (초/중/고 각각이 부동산 가치에 미치는 영향 차이)
- 변수 생성 시 고려한 한계와 대응:
  - **지오코딩 실패 20건(0.3%)**: 소규모 빌라/맨션 단지로, VWorld DB 누락. 거래 단위 0.13% 결측은 통계적으로 영향 미미
  - **환승역 중복**: 783행 중 128개 중복이지만 좌표가 거의 동일해 최소거리 산출 정확도엔 영향 없음
  - **단지 단위 좌표 한계**: 대단지의 경우 단지 내 동·층별 위치 차이는 무시됨 (모델 매크로 변수로는 충분)
- 효율성 관련 의사결정:
  - **거래 단위(10.6만) 대신 단지 unique(6,690) 지오코딩**: 같은 단지는 좌표 동일이라는 점을 활용하여 API 호출 16배 절약, 최종 단계에서 join으로 전체 거래에 매핑되므로 정보 손실 없음
  - **수도권 전체 지하철역 유지(서울만 거르지 않음)**: 외곽 단지의 실제 최근접 역이 경기·인천에 있는 경우 다수. 행정 경계 기준이 아닌 실제 접근성을 반영하기 위함

## 6. 주의사항 / 체크포인트

- ✅ 지오코딩 실패 건은 결측(NaN)으로 표시하여 전처리 파트에서 처리하도록 인계
- ✅ 거리 계산은 haversine 공식 사용 (단순 유클리드 거리는 위경도에서 부정확)
- ⚠️ **VWorld API 키는 GitHub에 업로드하지 않도록 주의** (.gitignore 처리, 또는 별도 환경변수)
- ⚠️ 코드의 `KAKAO_API_KEY`, `VWORLD_KEY` 변수는 키 노출 상태이므로 공유 전 반드시 제거
- ✅ 출처 URL과 다운로드 날짜는 보고서에 명시 필수
- ✅ 인코딩 차이(cp949, utf-8, utf-8-sig)는 코드에 명시되어 있으므로 재실행 시 문제 없음
